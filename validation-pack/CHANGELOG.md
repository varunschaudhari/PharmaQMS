# PharmaQMS Validation Pack — Change Log

Per-release change log with impact assessment (SPEC.md §5.5). One row per release; requirement IDs must match the traceability matrix.

| Date | Requirement IDs | Description | Impact |
|---|---|---|---|
| 2026-07-11 | N/A (SPEC §8 Phase 0, step 1 — repo scaffold) | Initial monorepo scaffold: npm workspaces (server/client/packages/shared), NestJS server shell with strict TS, Vite+React+Tailwind client shell, Jest+Supertest+mongodb-memory-server, Vitest+RTL, shared ESLint/Prettier config, docker-compose (Mongo+Redis). No business logic implemented. | new feature — no impact on existing validated functions (no prior release exists) |
| 2026-07-11 | PLT-1 | Auth & RBAC: tenant-scoped User/Role Mongoose schemas, login with JWT access+refresh tokens, refresh-token rotation (tokenVersion invalidation), configurable account lockout after N failed attempts, configurable password policy (complexity validator + expiry flag), permission-matrix enum (view/create/edit/review/approve/admin per module) with PermissionsGuard + @RequirePermission(), JwtAuthGuard/TenantGuard, global AppException/AllExceptionsFilter error shaping, ZodValidationPipe. Client: AuthContext, axios instance with refresh-on-401 interceptor, LoginPage, ProtectedRoute. 29 server tests + 3 client tests, all passing. | new feature — no impact on existing validated functions. Audit-trail hooks for failed-login/lockout events are stubbed with `TODO(PLT-2)` pending the audit service. |
| 2026-07-11 | PLT-2 (+ PLT-1 follow-up) | Audit Trail: append-only `auditEvents` Mongoose model (create/find only — every update/delete path rejected at the schema level via pre-hooks), field-level old→new diff utility, `AuditService.record()`, `@Audited()` decorator + `AuditTrailInterceptor` for declarative per-controller opt-in, paginated per-record history endpoint, CSV export per record and per module. Wired PLT-1's login success/failure/lockout events directly into the audit trail (pre-authentication boundary, so this bypasses the decorator pattern by design) and added a `POST /auth/change-password` endpoint (using the `@Audited()` pattern) so password changes are also audited, per this session's explicit scope. Client: shared `HistoryTab` component. 63 server tests + 4 client tests, all passing. | new feature — no impact on existing validated functions. `AuthService` now depends on `AuditService`; `PermissionsGuard`/history endpoints unchanged for existing routes. |
| 2026-07-11 | PLT-3 | E-Signatures: `POST /esign/challenge` re-verifies the user's password (tenant-configurable credential type; PIN documented as a future extension pending a `pinHash` field on User) and issues a short-lived (≤2 min, hard-capped in code), single-use signing token (JWT with a `jti` nonce). `SignatureGuard` verifies + atomically consumes that token (unique index on `jti` — a replay fails the insert) and rejects signing attempts backed only by a valid session. Append-only `signatures` collection (extracted the append-only Mongoose hook set from PLT-2's `auditEvents` into a shared `applyAppendOnly()` plugin, reused by both) storing user, UTC timestamp, meaning (`SignatureMeaning` enum: Reviewed by / Approved by / Trained — read and understood / Verified by / QA Disposition), entity ref, and a canonicalized SHA-256 hash of the entity snapshot at signing (`hashEntitySnapshot`/`verifyEntitySnapshot`, with tamper-detection tests). Client: shared `SignatureDialog` (credential re-entry, error handling) and `SignatureManifest` (name/meaning/timestamp) components. Also fixed a latent test-infra bug: RTL's auto-cleanup never registered between tests because the client's Vitest config uses `globals: false` — added explicit `afterEach(cleanup)` in setup-tests.ts. 91 server tests + 8 client tests, all passing. | new feature — no impact on existing validated functions. `auditEvents`'s append-only enforcement now runs through the shared plugin (behavior unchanged, existing tests still pass). |
| 2026-07-11 | PLT-5, PLT-8 | Numbering + Tenant/User Admin: this session finally introduces the `Tenant` entity itself (previously a bare `tenantId` with no backing document). **PLT-5** — tenant-configurable numbering schemes (prefix, department token, zero-padding, yearly reset) formatting as SOP-QA-001 / EQP-0042 / TRN-2026-0113; atomic gapless sequence generation via `findOneAndUpdate` `$inc`/upsert against a unique (tenantId, entityType, departmentCode, year) counter key, proven by a 50-concurrent-request test. Found and fixed a real (not just test) race: Mongoose builds indexes asynchronously in the background, so a brand-new counter's very first concurrent requests could beat the unique index into existence — fixed via an `onModuleInit` hook that awaits index creation before the app accepts traffic. **PLT-8** — `Tenant` schema with settings (timezone default Asia/Kolkata, e-sign credential type, session-timeout TTLs — JWT *secrets* remain env-only, never tenant-stored), `Department` master CRUD, `User` CRUD with role assignment, all audited via `@Audited()`. Tenant provisioning is platform-admin-only (`isPlatformAdmin` flag on User, orthogonal to the tenant-scoped permission matrix) via a new `PlatformAdminGuard`; the very first platform admin must be seeded directly into the database (documented operational bootstrap step, not an HTTP endpoint). `AuthService`/`EsignService` now resolve JWT-TTL/credential-type settings from the tenant document when present, falling back to the platform env defaults otherwise. Client: Tenants (super-admin only), Departments, Users, and Numbering Schemes admin pages, wired into nav/router with a `PlatformAdminRoute` guard. 120 server tests + 11 client tests, all passing. | new feature — no impact on existing validated functions. `AuthService`/`EsignService` constructors now also depend on the `Tenant` model (existing tests updated accordingly); User schema gained `isPlatformAdmin`/`departmentId` fields (both optional/defaulted, no migration needed). Password policy and lockout thresholds remain platform-env-configured, not yet tenant-configurable — narrower than earlier sessions' comments implied; flagged for a future session if needed. |
| 2026-07-11 | PLT-4 | Workflow Engine: generic, tenant-configurable approval engine usable by any business entity via a polymorphic `entityType`/`entityId` pair. `WorkflowTemplate` holds an ordered list of steps, each with a role-based assignee (`roleId`), a required `SignatureMeaning`, and a `rejectToStepIndex` (`null` = rejection returns the instance to the author's `DRAFT`; a number = re-enter `IN_PROGRESS` at that earlier step). `WorkflowInstance` tracks `status`/`currentStepIndex` and an optional `overrideAssigneeUserId` for reassignment. Actions — `submit`, `approve` (re-authenticated e-signature via PLT-3), `reject` (mandatory comment, zod `discriminatedUnion` enforces this per-action), `reassign` (admin-only, audited with a mandatory reason) — are dispatched through an explicit `WORKFLOW_ALLOWED_ACTIONS` transition map in `packages/shared`; invalid transitions throw before any state changes. Emits a `workflow.step-changed` event on every step change via `@nestjs/event-emitter`, with no listeners yet — reserved for PLT-6 Notifications. Endpoints: `GET /workflow/my-pending-tasks`, `GET /workflow/instances/:id`, `POST /workflow/instances/:id/act`, plus template CRUD. Refactored `SignatureGuard`/`EsignService` to expose `verifyAndConsumeSigningToken()` so the workflow engine can verify a single-use signing token in-process before writing its own `Signature` record (signatures are append-only, so a pre-existing signature can never be "attached" to a workflow after the fact). Extended `@Audited()`/`AuditTrailInterceptor` to accept a per-call `entityType`/`action` override, since one generic controller now writes audit events against many different business-entity types. Client: `PendingTasksPage`, `WorkflowStepper`, and a `WorkflowInstancePage` that reuses the (now control-inverted) `SignatureDialog` for approvals and a plain comment form for rejections. 142 server tests + 18 client tests, all passing (including full-happy-path with two real signatures, reject-back-and-reapprove, non-assignee-blocked, and reassign-audited-with-reason). Also fixed a Vite/Rollup production-build failure: Rollup's CJS-interop analysis can't statically resolve a plain `const` (`SIGNATURE_MEANING_LABELS`) through `packages/shared`'s multi-level `__exportStar` barrel re-exports, so the client now aliases `@pharmaqms/shared` straight to its TS source in `vite.config.ts` (the server is unaffected — it still consumes the compiled CJS `dist` via `require`). Live-verified end-to-end against the real Docker Mongo/Redis instances (not just automated tests): seeded a bootstrap platform admin, provisioned a real tenant, created real Dept Head/QA Head/Substitute users, and drove all three flows over real HTTP with real bcrypt-verified signature challenges. | new feature — no impact on existing validated functions. `SignatureGuard` behavior is unchanged from the caller's perspective (same request shape, same rejection cases); `@Audited()`'s new `entityType`/`action` override fields are optional and default to the decorator's static values, so all existing `@Audited()` call sites are unaffected. There is still no HTTP surface for creating `Role` documents beyond the one auto-created at tenant provisioning — additional roles (e.g. Dept Head, QA Head) must be seeded directly in the database until a future session adds role management endpoints; flagged for follow-up. |
